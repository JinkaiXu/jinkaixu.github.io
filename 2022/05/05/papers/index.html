    A Survey on Advancing the DBMS Query Optimizer: Cardinality Estimation, Cost Model, and Plan Enumeration

    2021

    由于不准确的基数预估、代价模型的错误和巨大的计划空间，查询优化器不能为一个复杂的查询在合理的时间内找到最优的执行计划。这篇文章首先介绍了上述问题产生的原因，接着又 review 了为提升三个关键部分（基数预估、代价模型、计划枚举）的质量的一些技术，最后提供了他们在以上各个方面未来方向的见解。

    理论上来说，在基数和代价估计准确的前提下，计划枚举可以在巨大的搜索空间上高效的进行。

    paper 的贡献如下：

    （1）总结了 cardinality estimation（CE），cost model（CM）和 plan enumeration（PE）做的不好的原因。

    （2）review 了用来让基数预估更准确已发表的工作，根据技术使用，将它们分为：基于概要的方法、基于采样的方法、基于学习的方法。

    （3）review 了在提升代价模型方面的工作，将它们分类为三组：改进现有模型、cost model 的代替方案、对一个简单查询的性能预测。

    （4）review 了在计划枚举中使用的技术并且 study 了处理大量查询的非学习的方法。还 review 了最近被提出的方法：采用强化学习来选择 join order。

    （5）分别提供了他们在以上各个方面未来方向的见解。

    为什么查询优化器的三大关键组件仍然不准确

    基数预估：

    （1）Error in single table with predications：DBMS 通常采用直方图来近似代表数据的分布，但是直方图的数据时小于原始数据的，并且直方图中也有一些假设：单个属性是均匀的，多个属性之间是独立的，一旦假设失效，就会导致错误。对于表中相关的属性，又提出了 Multi-histograms 来处理，但是这又会受到表大小的制约。

    （2）Error in multi-join queries：如果相关性存在于不同的表中，就没有有效的方法去获取两个或多个表之间的概要。join operator 的基数是采用 inclusion principle 通过它的孩子来计算的。如果假设有误就会产生很大的错误。除此之外，对于几个复杂查询，基数预估错误会从叶子结点传播到根节点，目前商用和开源的 DBMS 在多表连接基数预估时仍然很困难。

    （3）Error in user defined function：大多数 DBMS 支持 user-defined function(UDF)，当 UDF 存在于 condition 时，没有通用的方法来估计有多少 tuples 满足它。

    代价模型：

    影响的因素有：硬件、operator 的实现、operator 要处理的 tuple 的数量、当前数据库的状态（buffer 中的数据、并发的查询）、基数预估的准确性。如果是部署在分布式或并行的数据库，云环境或者跨平台的查询引擎，cost model 的复杂性会急剧增加，而且，即使基数预估是正确的，对一个查询的代价估计随着运行时间不是线性的，这可能会导致一个次优的查询计划。

    计划枚举：

    计划枚举有三个限制因素：（1）基数预估和代价模型的错误；（2）搜索空间中剪枝的策略；（3）当查询设计大量表时，优化器必须牺牲最优计划并且采用启发式算法来让优化时间变得合理。

    总结：

    基数预估的误差会传播到代价模型并且导致次优的 join order，消除或降低基数预估的误差是构造好的优化器的第一步。

    3. 基数预估

    3.1 基于概要的方法 Synopsis-based Methods：

    Histogram：

    统计直方图方面有三种工作：（1）快速构建；（2）新的 partition 方法提高准确度；（3）基于查询反馈的直方图构建

    方法：VODP -> AHistL（降低复杂度） -> GDY（基于 local search 的贪心算法，生成好的 sample boundaries 来在 VODP 中产生最优的 partition）-> GDY 的变体对比于 AHistL 在处理”效率-质量“的 trade-off 上更有优势 -> 采用在数据集上随机采样的方法而不是扫描整个数据集 -> 受到采样的启发，提出 merging 算法。

    Sketch：

    Sketch 将列建模为向量或矩阵，以计算值上元组的不同计数或频率（例如：Count Min）。用不同的 sketches 来估计 join size，这个工作是不考虑 filter 的，基本的 idea 是：（1）在 join attribute 建立 sketch 并且忽略其他所有属性；（2）根据 vector 或者 matrices 相乘估计 join size。这个方法只支持等值连接和单个column的连接。在 sketch 中引入一个 filter 的可能方法是构建一个假想表，这个表仅包含满足 filter 的元组的连接值。

    Other Techniques：

    TuG 是一个基于图的概要，TuG 的 node 表示同一个表的 tuple 的集合或者同一个 attribute 的 value 的集合，边表示不同表或 attribute 之间的 join 关系。但是构建速度太慢，无法在大型 dataset 中使用。

    3.2 基于采样的方法 Sampling-based Methods：

    synopsis-based 的方法相当难以捕获不同表之间的相关性。一些人尝试从表中采样，然后将查询在样本上运行来进行基数预估，只要获得的样本的数据分布接近原始数据，那么基数预估就是可信的。

    3.3 基于学习的方法 Learning-based Methods：

    采用基于学习的方法来捕获数据的分布的相关性。

    Cardinality Estimation in DBMS: A Comprehensive Benchmark Evaluation

    1. 提出 STATS-CEB，主要用来评估那些 ML-based 的基数预估方法

    2. 认为广泛使用的 Q-Error 无法区分不同子计划查询的重要性，因此无法无法真实反映基数预估方法生成的查询计划质量，于是提出了一个新的 metric P-Error 去评估基数预估方法的性能，它克服了 Q-Error 的限制，能够反映基数预估方法的整体端到端性能。

    Background：

    开源数据库的基数预估主要由两种：histogram ：PG、SQL Server，sampling：MySQL、MariaDB。基数预估的核心工作是构建一个压缩的 model 来捕获 data 或 query 的信息。基于学习的方法可以被分为两类：data-driven 和 query-driven，query-driven 的方法构建有辨别力的方法来将有特点的 query 映射到它们的基数；data-driven 的方法直接对所有属性的 joint distribution 进行建模。与传统方法相比，他们的估计准确性更加突出因为它们更加复杂和细粒度的。

    Motivation：

    尽管基数预估方法取得了最新进展，但忽视了一个基础问题，这些先进的基数预估方法在真实世界的设定中能多大程度提高查询优化器的性能？尽管现有的研究已经进行了广泛的实验，他们有以下缺点：

    （1）用于评估的数据和查询负载可能不能很好的反映真实世界的场景。

    （2）大多数评估没有展示基数预估方法在查询优化器上的端到端的提升。
